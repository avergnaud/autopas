# Prompt 26

Analyse le code existant, en particulier les fonctions d'anonymisation.
Si l'utilisateur définit ces 2 remplacements :
Enron -> CLIENT
Enron Groupe -> CLIENT

Est-ce que ça va fonctionner ?

## Réponse

Le tri par longueur décroissante est bien présent (ligne 194-199 pour xlsx, 322-327 pour docx). Pour l'anonymisation, ça fonctionnerait correctement :
1. `Enron Groupe` (12 chars) remplacé en premier → `CLIENT`
2. `Enron` (5 chars) remplacé ensuite → `CLIENT`

Mais la dé-anonymisation serait cassée.

Dans `deanonymize_xlsx`, le dict comprehension `{token: original ...}` ne peut avoir qu'une seule clé `CLIENT`, donc l'une des deux origines écrase l'autre. Tous les `CLIENT` seraient restaurés en `Enron`, y compris ceux qui étaient `Enron Groupe`.

Solution recommandée : utiliser des tokens distincts (ex: `CLIENT_GROUPE` et `CLIENT`).

## Correctif — Verbosité dans le wizard corpus

### Problème

Le wizard "Base de connaissances" utilisait le même endpoint `/api/questions` que le
cadrage PAS, et posait donc la question de verbosité à l'utilisateur lors de l'upload
d'un fichier corpus. La verbosité n'a pas de sens à ce stade : elle pilote la génération
des réponses au questionnaire, pas l'indexation d'un fichier de référence.

### Fix

`web/private.html` — ligne 1059 : filtrer la question `verbosity` avant de construire
le wizard corpus :

```js
questions: qData.questions.filter(q => q.key !== 'verbosity'),
```

La question de verbosité reste présente dans le wizard de cadrage PAS (inchangé).

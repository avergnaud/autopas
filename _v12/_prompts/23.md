# Prompt 23 — Remplissage automatique de la colonne "status" par Claude

## Contexte

Les questionnaires xlsx contiennent souvent une colonne "statut" (ex : "Statut", "Compliance",
"Status") avec une liste déroulante (dropdown) proposant quelques choix fixes, typiquement :

- `Conforme` / `Non conforme` / `Partiel` / `N/A`
- `Oui` / `Non` / `Partiel`
- `Compliant` / `Non-Compliant` / `Partial` / `N/A`
- etc.

La colonne est déjà détectée lors de l'analyse de structure (`col_status` dans `structure.json`).
Actuellement, cette colonne est identifiée mais jamais remplie lors de la génération.

## Objectif

Lorsque `col_status` est présent dans la structure et que des choix de dropdown sont lisibles
dans le fichier xlsx, Claude doit — en plus de rédiger la réponse textuelle — sélectionner la
valeur de statut la plus appropriée parmi les choix disponibles.

## Étape 1 — Lire les choix du dropdown dans parser_xlsx.py

Ajouter une fonction `read_status_choices(xlsx_path: Path, structure: dict) -> list[str]`.

### Logique

1. Si `structure.get("col_status")` est `None` → retourner `[]`
2. Charger le workbook avec openpyxl (read_only=False pour accéder aux data_validations)
3. Identifier la feuille principale (`selected_sheet`)
4. Parcourir `ws.data_validations.dataValidation` :
   - Pour chaque `dv` avec `dv.type == "list"` :
     - Vérifier que `dv.sqref` (ou `str(dv.sqref)`) contient la lettre de colonne du statut
     - **Case 1** — `formula1` est une liste inline (commence et finit par `"`) :
       - Retirer les guillemets et splitter sur `,` → liste de choix
     - **Case 2** — `formula1` est une référence directe (ex: `SheetName!$A$1:$A$5`) :
       - Tenter de lire les cellules de cette plage dans le workbook → liste de choix
     - **Case 3** — `formula1` est un nom nu (ex: `Cotation`) — named range :
       - openpyxl perd les defined names locaux quand un homonyme workbook-level `#REF!`
         les écrase (même bug que `safe_local_defined_names`)
       - Résoudre via raw XML avec `_resolve_named_range_from_xml()` : lire
         `xl/workbook.xml` directement (zipfile + ElementTree), chercher le `<definedName>`
         correspondant en préférant `localSheetId == index_feuille_courante`
       - En déduire la référence réelle (ex: `'PESA SSI Presta FR'!$A$7:$A$10`) et
         lire les cellules → liste de choix
     - Retourner la liste (strippée, filtrée des vides) dès qu'on trouve un match
5. Si aucune data_validation trouvée → retourner `[]`

### Signature

```python
def read_status_choices(xlsx_path: Path, structure: dict) -> list[str]:
    """Read dropdown choices for the status column from xlsx data validations.

    Args:
        xlsx_path: Path to the xlsx file.
        structure: Structure dict with col_status and selected_sheet.

    Returns:
        List of valid status strings, or [] if not found or not applicable.
    """
```

### Détail : vérifier que la validation s'applique à la bonne colonne

`dv.sqref` est un objet `CellRange` ou une chaîne. Le convertir en string et vérifier
que la lettre de colonne de `col_status` y apparaît. Exemple :

```python
col_status_letter = structure["col_status"].upper()
sqref_str = str(dv.sqref)
# sqref_str ressemble à "F2:F200" ou "$F$2:$F$200"
import re
if re.search(rf'\b{col_status_letter}\d', sqref_str.replace('$', '')):
    # cette validation s'applique à la colonne status
```

## Étape 2 — Modifier system_response.txt

Ajouter une section conditionnelle dans le format de sortie JSON.

Le champ `status` est optionnel : il ne doit apparaître dans la réponse que si des
choix sont fournis dans le prompt utilisateur.

### Nouvelle section dans FORMAT DE SORTIE

```
FORMAT DE SORTIE :
Retourner un JSON valide avec la structure :
{
  "responses": [
    {"question_id": "...", "response": "...", "status": "..."},
    ...
  ]
}

Le champ "status" :
- N'est requis QUE si une liste de valeurs autorisées est fournie dans le prompt
  (section "=== VALEURS DE STATUT AUTORISÉES ===").
- Doit être exactement l'une des valeurs de cette liste (correspondance exacte, respecter
  la casse).
- Logique de choix :
  * "Conforme" / "Oui" / "Compliant" → la prestation répond pleinement à l'exigence.
  * "Non conforme" / "Non" / "Non-Compliant" → la prestation ne satisfait pas l'exigence,
    ou l'exigence est explicitement rejetée.
  * "Partiel" / "Partial" → la prestation satisfait partiellement, ou une réserve est émise.
  * "N/A" / "Non applicable" → l'exigence ne s'applique pas au périmètre de la prestation
    (la réponse commence par "Non applicable...").
  * En cas de doute entre Conforme et Partiel, préférer Partiel (prudence contractuelle).
- Si la liste est absente du prompt, omettre le champ "status" de chaque réponse.
```

## Comportement attendu (cas nominaux)

| Situation | Comportement |
|---|---|
| `col_status` absent de la structure | Aucun changement, pas de champ `status` dans le JSON Claude |
| `col_status` présent, dropdown lisible | Les choix sont injectés dans le prompt, Claude retourne `status` pour chaque réponse, la valeur est écrite dans le fichier output |
| `col_status` présent, dropdown illisible (range externe) | `read_status_choices` retourne `[]`, comportement identique au cas "absent" |
| Claude retourne un `status` invalide (hors liste) | Valeur écrite telle quelle dans le xlsx (Claude est guidé pour respecter la liste exacte) |

## Bug découvert lors du test (QUESTIONNAIRE FOURNISSEUR AESIO MUTUELLE 2026.xlsx)

Le `formula1` de la data validation sur col F était `Cotation` (named range, Case 3).
La définition locale dans le XML était `localSheetId=0 → 'PESA SSI Presta FR'!$A$7:$A$10`
mais openpyxl la lisait `#REF!` à cause d'un homonyme workbook-level sans localSheetId.
Sans le Case 3, `read_status_choices` retournait `[]` et la colonne n'était jamais remplie.

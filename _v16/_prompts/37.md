# Prompt 37 — Exclusions de périmètre dérivées du cadrage

## Contexte et problème observé

En comparant un questionnaire rempli par un humain (`AVE.QUESTIONNAIRE FOURNISSEUR AESIO MUTUELLE 2026.xlsx`) avec le questionnaire rempli par notre application (`PAS_rempli_d245d6e1.xlsx`), on constate **46 différences sur 98 lignes dans la colonne statut (F)**.

La quasi-totalité (40/46) suit le même pattern :

- **Humain** : `N/A`
- **Application** : `2 -Totalement conforme` ou `1 - Partiellement conforme`

Les sections concernées sont :
- **C** — Sécurité de l'exploitation du service (10 questions)
- **D** — Sécurité de l'hébergement et des infrastructures (12+ questions)
- **E** — Sécurité des accès réseau (2 questions)
- **A 3.4, A 3.6** — Gouvernance / comité de suivi
- **B 1.1, B 1.5** — Auditabilité

Le cadrage de ce projet est :
```json
{
  "hebergement_donnees": "SI CLIENT",
  "poste_travail": "CLIENT",
  "type_prestation_base": "Assistance Technique"
}
```

Ces valeurs signifient clairement que **FOURNISSEUR n'héberge rien, n'opère aucune infrastructure, et n'administre aucun poste de travail**. Toutes les questions relatives à ces périmètres sont hors scope et auraient dû recevoir le statut `N/A`.

## Cause racine

Le fichier `data/config/prompts/system_response.txt` contient déjà une section "RÈGLE D'APPLICABILITÉ" qui liste des exemples génériques :

```
- Question sur l'hébergement Cloud → non applicable si hebergement_donnees = "SI CLIENT"
- Question sur les postes FOURNISSEUR → non applicable si poste_travail = "CLIENT"
- ...
```

Et la fonction `_build_user_prompt_responses()` dans `response_generator.py` injecte le cadrage brut dans le prompt via `_format_cadrage(cadrage)` :

```
=== CONTEXTE DE LA PRESTATION ===
- Hébergement des données : SI CLIENT
- Type de poste de travail : CLIENT
- ...
```

**Le problème** : Claude reçoit les données du cadrage + des règles génériques d'exclusion, mais doit lui-même effectuer le raisonnement "cadrage X → questions Y hors périmètre" pour chaque question, questionnaire par questionnaire. En pratique, Claude ne tient pas suffisamment compte de ces règles et répond aux questions hors périmètre comme si elles étaient applicables.

**La solution manquante** : dériver explicitement les exclusions de périmètre à partir des valeurs du cadrage, et les injecter dans le prompt utilisateur sous forme de déclarations claires et actionnables, question par question.

## Solution à implémenter

### 1. Nouvelle fonction `_derive_scope_exclusions()` dans `response_generator.py`

Créer une fonction qui traduit les valeurs du cadrage en **déclarations d'exclusion explicites** en français naturel :

```python
def _derive_scope_exclusions(cadrage: dict, status_choices: list[str] | None) -> str | None:
    """Dérive les exclusions de périmètre depuis le cadrage.

    Retourne un bloc de texte multi-lignes à injecter dans le prompt,
    ou None si aucune exclusion n'est applicable.
    """
```

La fonction doit couvrir les cas suivants :

| Valeur du cadrage | Exclusion à déclarer |
|---|---|
| `hebergement_donnees == "SI CLIENT"` | L'hébergement, l'infrastructure serveur, le réseau d'infrastructure, l'exploitation du SI (CMDB, patchs serveurs, sauvegardes, logs système, cloisonnement réseau, firewalls, IDS/IPS, environnements hors-prod côté hébergement) sont gérés par CLIENT. Ces domaines sont **hors périmètre FOURNISSEUR**. |
| `poste_travail == "CLIENT"` | Les postes de travail sont fournis et administrés par CLIENT (MDM, antivirus poste, chiffrement disque, gestion des mises à jour poste, politique de verrouillage). Ces sujets sont **hors périmètre FOURNISSEUR**. |
| `sous_traitance_rgpd == "Non"` | FOURNISSEUR n'a pas de sous-traitants RGPD pour cette prestation. Les questions sur la gestion des sous-traitants sont **hors périmètre**. |
| `"Développement"` absent de `activites` | FOURNISSEUR ne réalise pas de développement logiciel dans le cadre de cette prestation. Les questions sur le cycle de vie sécurisé du développement (SSDLC, tests de sécurité applicatifs, revue de code) sont **hors périmètre**. |
| `"Agence FOURNISSEUR"` absent de `lieu_travail` | FOURNISSEUR n'intervient pas depuis ses propres locaux. Les questions sur la sécurité physique des locaux FOURNISSEUR sont **hors périmètre**. |

Pour chaque exclusion détectée, la fonction précise aussi **quelle valeur de statut utiliser** :

```
→ Si "N/A" est disponible dans les valeurs autorisées : utiliser "N/A".
→ Si une valeur équivalente existe (ex: "Non applicable", "NA") : l'utiliser.
→ Si aucune valeur N/A n'est disponible mais qu'une colonne statut existe : laisser le statut vide (ne pas mettre "0 - Non implémenté").
→ Si pas de colonne statut : indiquer "Sans objet — [raison courte]" en début de réponse.
```

La détection de la valeur N/A disponible se fait en cherchant parmi `status_choices` une valeur dont la version lowercase contient `"n/a"`, `"non applicable"`, `"na"` ou équivalent.

### 2. Injection dans `_build_user_prompt_responses()`

Après la section `=== CONTEXTE DE LA PRESTATION ===` et avant `=== EXEMPLES DE QUESTIONNAIRES DÉJÀ REMPLIS ===`, ajouter :

```python
exclusions = _derive_scope_exclusions(cadrage, status_choices)
if exclusions:
    lines.append("=== EXCLUSIONS DE PÉRIMÈTRE ===")
    lines.append(
        "Sur la base du cadrage ci-dessus, les domaines suivants sont HORS PÉRIMÈTRE "
        "pour cette prestation. Pour toute question relevant de ces domaines, appliquer "
        "les consignes de statut indiquées."
    )
    lines.append(exclusions)
    lines.append("")
```

### 3. Mise à jour de `system_response.txt`

La section "RÈGLE D'APPLICABILITÉ" existante reste utile comme filet de sécurité, mais son rôle change : elle ne doit plus être la *seule* source de vérité sur les exclusions. Elle devient un complément pour les cas non couverts par les exclusions explicites.

Ajouter une note en tête de cette section :

```
RÈGLE D'APPLICABILITÉ :
Les exclusions de périmètre spécifiques à cette prestation sont listées dans la
section "=== EXCLUSIONS DE PÉRIMÈTRE ===" du message utilisateur (si présente).
Ces exclusions sont prioritaires. La présente règle s'applique en complément
pour les cas non listés.
```

## Exemple de sortie attendue pour le cadrage AESIO

Pour le cadrage `hebergement_donnees = "SI CLIENT"`, `poste_travail = "CLIENT"`, `sous_traitance_rgpd = "Non"`, `lieu_travail = ["Site CLIENT", "Télétravail"]`, la section générée serait :

```
=== EXCLUSIONS DE PÉRIMÈTRE ===
Sur la base du cadrage ci-dessus, les domaines suivants sont HORS PÉRIMÈTRE
pour cette prestation. Pour toute question relevant de ces domaines, appliquer
les consignes de statut indiquées.

1. HÉBERGEMENT ET INFRASTRUCTURE (hebergement_donnees = "SI CLIENT")
   CLIENT héberge et opère l'intégralité de l'infrastructure. FOURNISSEUR n'administre
   aucun serveur, aucun réseau d'infrastructure, aucun environnement d'hébergement.
   Questions hors périmètre : hébergement, serveurs, CMDB, patchs système, sauvegardes,
   logs d'infrastructure, cloisonnement réseau, firewalls, IDS/IPS, accès d'exploitation,
   gestion des comptes administrateurs d'infrastructure, environnements hors-prod côté
   hébergement, audits techniques du SI hébergé.
   → Statut : utiliser "N/A" (valeur disponible dans ce questionnaire).

2. POSTES DE TRAVAIL (poste_travail = "CLIENT")
   Les postes de travail sont fournis et administrés par CLIENT. FOURNISSEUR n'est
   responsable d'aucune politique de sécurité poste.
   Questions hors périmètre : MDM, antivirus poste, chiffrement disque, politique de
   mots de passe poste, gestion des mises à jour poste, verrouillage automatique.
   → Statut : utiliser "N/A" (valeur disponible dans ce questionnaire).

3. SOUS-TRAITANCE RGPD (sous_traitance_rgpd = "Non")
   FOURNISSEUR n'a pas de sous-traitants RGPD pour cette prestation.
   Questions hors périmètre : gestion contractuelle des sous-traitants, audits des
   sous-traitants, clauses RGPD sous-traitants.
   → Statut : utiliser "N/A" (valeur disponible dans ce questionnaire).

4. LOCAUX FOURNISSEUR (lieu_travail ne comprend pas "Agence FOURNISSEUR")
   FOURNISSEUR n'intervient pas depuis ses propres locaux.
   Questions hors périmètre : sécurité physique des locaux FOURNISSEUR, badges d'accès,
   surveillance des locaux, destruction des supports en agence.
   → Statut : utiliser "N/A" (valeur disponible dans ce questionnaire).
```

## Cas limites à gérer

- **Pas de colonne statut** : `status_choices` sera `None` ou `[]`. La section d'exclusions indique "écrire 'Sans objet — [raison]' au début de la réponse".
- **Colonne statut sans valeur N/A** : la section indique "laisser le statut vide" (ne pas forcer `0 - Non implémenté` qui serait trompeur).
- **`pas_niveau_entreprise == "Oui"`** : dans ce cas, ne générer aucune exclusion (toutes les questions sont applicables — déjà documenté dans `system_response.txt`).
- **`hebergement_donnees == "Cloud"`** : FOURNISSEUR héberge dans le cloud mais ne gère pas l'infrastructure physique. L'exclusion est partielle : les questions sur le datacenter physique sont N/A, mais les questions sur la sécurité applicative et la configuration cloud restent applicables. Traiter avec nuance dans la description.

# Prompt 28

## Feature : Injection de POLITIQUES.md dans le prompt Claude + règles d'applicabilité

### Contexte

- `data/policies/politiques.md` — fichier unique uploadé par l'administrateur (via l'onglet "Politiques de sécurité", prompt 27).
- Ce fichier décrit les politiques de sécurité internes du FOURNISSEUR (PSSI, charte, procédures...).

### Changements à implémenter

#### 1. Cache du fichier via l'API Files d'Anthropic (Files API beta)

- À chaque génération, vérifier si `data/policies/politiques.md` existe.
- Uploader le fichier via `client.beta.files.upload(...)` et récupérer un `file_id`.
- Mettre en cache le `file_id` dans `data/policies/file_id_cache.json` avec le hash MD5 du contenu.
- Si le fichier n'a pas changé (même hash), réutiliser le `file_id` en cache.
- Passer le `file_id` à l'appel `client.messages.create(...)` via le champ `documents` (format `{"type": "document", "source": {"type": "file", "file_id": "..."}}`).

#### 2. Mise à jour du system prompt (`data/config/prompts/system_response.txt`)

Ajouter les règles suivantes dans le prompt système :

**Priorité des sources d'information (ordre décroissant) :**
1. Les métadonnées de cadrage (type de prestation, poste de travail, hébergement, etc.) — source la plus importante, elle définit le périmètre réel de la prestation.
2. Les fichiers de référence du corpus (exemples de questionnaires déjà remplis) — à utiliser pour le style et le contenu des réponses.
3. Le document POLITIQUES.md joint — à utiliser quand la question est applicable ET que POLITIQUES.md est fourni.

**Règle d'applicabilité :**
- Pour chaque question, déterminer si elle est dans le périmètre de la prestation (en se basant sur les métadonnées de cadrage).
- Si la question n'est **pas applicable** au périmètre (ex : question sur l'hébergement Cloud alors que les données sont sur le SI CLIENT, question sur les postes FOURNISSEUR alors que les collaborateurs utilisent des postes CLIENT, etc.) → répondre `"Non applicable sur le périmètre de la prestation"` suivi d'une justification courte (1 phrase), et choisir le statut N/A.
- Si `pas_niveau_entreprise = Oui` (questionnaire portant sur l'ensemble de l'entreprise FOURNISSEUR), alors **toutes** les questions sont considérées applicables, et POLITIQUES.md est systématiquement utilisé.
- Sinon (questionnaire de prestation), utiliser POLITIQUES.md uniquement pour les questions applicables.

#### 3. Mise à jour de `response_generator.py`

- Nouvelle fonction `_get_policies_file_id(client)` :
  - Lit `data/policies/politiques.md`.
  - Calcule le MD5 du contenu.
  - Compare avec le cache `data/policies/file_id_cache.json` (`{"md5": "...", "file_id": "..."}`).
  - Si match → retourner le `file_id` en cache.
  - Sinon → uploader via Files API, sauvegarder le cache, retourner le nouveau `file_id`.
  - Si le fichier n'existe pas → retourner `None`.
- Modifier `_call_claude_json()` pour accepter un `file_id: str | None` optionnel et l'inclure dans le message si fourni.
- Modifier `_do_generation()` pour appeler `_get_policies_file_id()` et passer le `file_id` à l'appel de génération des réponses.
- Modifier `_build_user_prompt_responses()` pour injecter dans le prompt la mention du document POLITIQUES.md et la valeur de `pas_niveau_entreprise` afin que Claude sache s'il est en mode "entreprise entière" ou "périmètre prestation".

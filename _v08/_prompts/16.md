# Prompt 16 — Deuxième crash OOM : ajustement des limites et optimisation openpyxl

## Logs du crash

```
Mar 08 08:32:39 appsec systemd[1]: pas-assistant.service: A process of this unit has been killed by the OOM killer.
memory: usage 1433600kB, limit 1433600kB   (MemoryMax=1400M atteint)
swap:   usage 524172kB,  limit 524288kB    (MemorySwapMax=512M atteint)
Out of memory: Killed process 202534 (uvicorn) total-vm:1923276kB, anon-rss:1322268kB
```

Stack trace : `do_swap_page → swapin_readahead → mem_cgroup_out_of_memory`
Warning openpyxl : `DrawingML support is incomplete... Shapes and drawings will be lost.`

## Analyse

- Les limites cgroup fonctionnent : c'est désormais un OOM cgroup (contrôlé), non plus un OOM kernel.
- uvicorn consomme ~1.32 GB RAM + ~409 MB swap = ~1.73 GB, dépassant les deux limites.
- Le fichier traité contient des drawings/images (DrawingML warning), ce qui aggrave la consommation openpyxl.

## Actions

### 1. Relâchement des limites cgroup (`pas-assistant.service.j2`)

```ini
MemoryMax=1700M
MemorySwapMax=1500M
```

Cela donne ~1.7 GB RAM + ~1.5 GB swap = ~3.2 GB virtuel pour uvicorn,
en laissant ~300 MB RAM et ~500 MB swap pour l'OS.

### 2. Réduction mémoire openpyxl (`anonymizer.py`, `web.py`)

Ajout de `rich_text=False` dans tous les `load_workbook()` en mode écriture :

```python
wb = load_workbook(source_path, keep_links=False, rich_text=False)
```

openpyxl n'a pas d'option `keep_drawings=False`, mais `rich_text=False` réduit
la mémoire allouée pour les cellules à contenu enrichi.

## Diagnostic ouvert

La consommation ~1.7 GB reste anormalement élevée. Pistes à investiguer :
- Taille et complexité du fichier xlsx traité (onglets, images, formules)
- Mémoire non libérée entre requêtes (comportement allocateur Python)
- Réponses Claude volumineuses conservées en mémoire

Solution durable : profiler avec `memory_profiler` ou upgrader le droplet à 4 GB RAM.

# Prompt 14 — Fix: xlsx corrompu à l'ouverture dans Excel

## Problème rapporté

Après génération, Excel affiche : "Désolé... Nous avons trouvé un problème dans le contenu de « PAS_rempli_proj_...xlsx »"

## Cause identifiée

Deux bugs dans la manipulation XML directe des fichiers xlsx (ZIP) :

### Bug 1 — `apply_deanonymization` : XML invalide après remplacement

La fonction `anonymizer.deanonymize()` fait un simple `text.replace(alias, real_name)` sur le XML brut. Si le nom réel contient `&` (ex : "Ministère des Armées & Cie"), le XML résultant contient un `&` non échappé → XML invalide → Excel repair dialog.

**Exemple :**
- Avant : `<t>Le CLIENT est responsable</t>`
- Après (bugué) : `<t>Le Ministère des Armées & Cie est responsable</t>` ← `&` invalide en XML
- Après (corrigé) : `<t>Le Ministère des Armées &amp; Cie est responsable</t>`

### Bug 2 — `apply_anonymization` : noms XML-encodés non trouvés

Si le nom réel contient `&`, il est stocké comme `&amp;` dans le XML de l'xlsx. Le pattern regex `re.escape("A & B")` ne trouve pas `A &amp; B` → la valeur n'est pas anonymisée → données réelles envoyées à l'API Claude (fuite potentielle).

## Correction apportée

Fichier modifié : `app/services/parser_xlsx.py`

Ajout de `import re` (manquant) et remplacement des deux appels directs par des fonctions XML-conscientes :

- **`_make_xml_anonymizer(anonymizer)`** : cherche à la fois le nom brut ET la version XML-encodée (`xml.sax.saxutils.escape(key)`)
- **`_make_xml_deanonymizer(anonymizer)`** : insère le nom réel XML-échappé (`saxutils.escape(real_name)`) au lieu de la version brute

Ces fonctions sont utilisées dans `apply_anonymization` et `apply_deanonymization` à la place de `anonymizer.anonymize` / `anonymizer.deanonymize`.

Le `anonymizer.py` reste intact (il opère sur du texte brut, c'est correct pour l'usage non-XML comme les prompts Claude).

# Prompt 19 — Fix : Excel repair dialog (Bug 4 — externalLinkLongPath)

## Contexte

Le dialogue Excel « Nous avons trouvé un problème dans le contenu de `PAS_rempli_*.xlsx` »
persistait malgré les fixes des prompts 15-18 (sqRef, externalLink rId, VML namespaces).

## Méthode de diagnostic

Ajout de snapshots côté serveur dans `response_generator.py` pour capturer l'état du fichier
xlsx après chaque étape du pipeline :

```
data/projects/{id}/debug/
  01_after_copy.xlsx          ← copie de original.xlsx (aucune modif)
  02_after_delete_sheets.xlsx ← après suppression des onglets exclus
  03_after_anonymization.xlsx ← après anonymisation ZIP/XML
  04_after_write_responses.xlsx ← après openpyxl.save() + fix_openpyxl_xml
  05_after_deanonymization.xlsx ← fichier final (= output.xlsx)
```

Fonction ajoutée dans `response_generator.py` :

```python
def _snapshot(src: Path, project_id: str, label: str) -> None:
    from app.services.project_manager import PROJECTS_DIR
    debug_dir = PROJECTS_DIR / project_id / "debug"
    debug_dir.mkdir(parents=True, exist_ok=True)
    shutil.copy2(src, debug_dir / label)
```

## Analyse des snapshots

### Résultats du scan XML

| Snapshot | Problème détecté |
|---|---|
| 01, 02, 03 | `sqref=` lowercase dans `<dataValidation>` → dialog sur l'ORIGINAL aussi |
| 04, 05 | `sqRef=` corrigé ✓ — mais dialog persiste → autre cause |

### Fichiers supprimés par openpyxl (76 → 38 entrées ZIP)

openpyxl supprime lors de la sauvegarde :
- `customXml/` (3 items + rels) — données XML personnalisées
- `xl/drawings/drawing1.xml` + `xl/media/image1.png` (image 248 KB)
- `xl/comments1.xml` → remplacé par `xl/comments/comment1.xml`
- `xl/drawings/vmlDrawing1.vml` → remplacé par `xl/drawings/commentsDrawing1.vml`
- `xl/calcChain.xml`, `xl/sharedStrings.xml`, tous les onglets supprimés par l'utilisateur

Aucune référence orpheline n'a été trouvée dans les rels — openpyxl met bien à jour les rels.

### Cause racine identifiée — externalLinkLongPath

Le fichier original avait pour **externalLink1** et **externalLink10** deux relations :

```xml
<!-- Original: externalLink1.xml.rels -->
<Relationship Id="rId1" Type="...externalLinkPath"
    Target="file:///\\C3B2D8ED\PESA%202022%20v1...xlsx" TargetMode="External"/>
<Relationship Id="rId2" Type=".../externalLinkLongPath"
    Target="https://catamaniacrm.sharepoint.com/..." TargetMode="External"/>
```

`externalLink1.xml` référence `r:id="rId1"`.

Après sauvegarde openpyxl, **seule la relation `externalLinkLongPath` est conservée**
et renommée `rId1` :

```xml
<!-- Après openpyxl + _fix_extlink_rels -->
<Relationship Id="rId1" Type=".../externalLinkLongPath"
    Target="https://catamaniacrm.sharepoint.com/..."/>
```

L'ID `rId1` correspond maintenant à une relation de type **`externalLinkLongPath`**
(type Microsoft propriétaire 2019+). Or `<externalBook r:id="rId1">` attend un
type **`externalLinkPath`** (standard OOXML). Excel détecte cette incohérence et
déclenche le dialogue de réparation.

**Note** : `_fix_extlink_rels` (existant) ne corrigeait pas ce cas, car après openpyxl
l'ID `rId1` existe bien dans le rels — seul le TYPE est wrong. Le fix existant ne
corrige que les mismatches d'ID.

## Correction

Fichier modifié : `app/services/parser_xlsx.py`

### Nouvelle fonction `_fix_extlink_longpath(filepath)`

```python
def _fix_extlink_longpath(filepath: Path) -> None:
    _LONG_PATH_TYPE = (
        "http://schemas.microsoft.com/office/2019/04/relationships/externalLinkLongPath"
    )
    _STD_PATH_TYPE = (
        "http://schemas.openxmlformats.org/officeDocument/2006/relationships/externalLinkPath"
    )
    # Pour chaque externalLink rels qui n'a QUE externalLinkLongPath (sans externalLinkPath) :
    # remplace le type URI pour le rendre standard.
    # Les rels qui ont déjà externalLinkPath ne sont pas touchés.
```

Condition de déclenchement : le rels contient `externalLinkLongPath` ET ne contient PAS `externalLinkPath`.

### Appel dans `fix_openpyxl_xml`

```python
_xlsx_zip_replace(filepath, _transform)   # Bug 1: sqref → sqRef
_fix_extlink_rels(filepath)               # Bug 2: externalLink rId mismatch
_fix_extlink_longpath(filepath)           # Bug 4: externalLinkLongPath seul ← nouveau
_fix_vml_namespaces(filepath)             # Bug 3: VML namespace prefix inconsistency
```

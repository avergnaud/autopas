# Prompt 13 — Bug fixes et améliorations

Pour cette itération, focus sur les corrections de bugs et améliorations identifiées après revue du code de _v06.

## Bugs à corriger

**Bug #1 — `parser_xlsx.py` : `write_responses` utilise `ws.max_row`**
`extract_questions` utilise correctement `iter_rows()` avec un stop sur les lignes vides consécutives, mais `write_responses` boucle sur `range(first_data_row, ws.max_row + 1)`. Pour les xlsx avec un tag `<dimension>` périmé (courant), `ws.max_row` peut renvoyer 1 048 576 — boucle quasi-infinie. Migrer vers `iter_rows()` avec le même mécanisme d'arrêt.

**Bug #2 — `parser_docx.py` : question incorrecte lors de l'extraction dans les tableaux**
Quand le marqueur de réponse est trouvé dans un tableau, la question est définie à `para.text.strip()` qui est le texte du marqueur lui-même, pas la question. Il faut chercher le texte de la question dans les cellules/paragraphes précédents de la même ligne ou de la ligne précédente.

**Bug #3 — `reference_selector.py` : fichiers de référence renvoyés même avec score 0**
Si aucun fichier du corpus n'est pertinent (tous à score 0), les `max_files` fichiers sont quand même envoyés à Claude comme "références". Du bruit inutile. Ajouter un seuil minimum (score > 0) pour exclure les fichiers non pertinents.

**Bug #8 — `reference_selector.py` : scoring toujours 0 (clés numériques vs clés sémantiques)**
Le cadrage est stocké avec des clés numériques (`"1"`, `"6"`…) mais `_score()` attend des clés sémantiques (`"type_prestation"`, `"hebergement_donnees"`…). Score toujours 0 → Bug #3 filtrait alors tout le corpus. Fix en 3 fichiers : directive `KEY:` dans `questions.txt`, parsée dans `config.py`, et `_translate_cadrage()` dans `reference_selector.py` qui traduit le dict avant scoring.

**Bug #7 — `claude_client.py` : `JSONDecodeError` sur réponse tronquée (constaté en production)**
`generate_attention_points` avait `max_tokens=4000` codé en dur. Sur un questionnaire de 129 questions, Claude tronquait sa réponse JSON (`Unterminated string`, char 13277), faisant planter tout le pipeline. Fix : utiliser `config["claude"]["max_tokens"]` (16 000). En bonus, `_parse_json` logue désormais le `stop_reason` et les 500 derniers caractères de la réponse brute en cas d'échec.

## Améliorations

**Amélioration #4 — `response_generator.py` : `_format_cadrage` avec clés numériques**
Le contexte cadrage envoyé à Claude contient les clés brutes du dict (ex. `"1 : Dispositif à engagement"`) au lieu du texte de la question. Enrichir `_format_cadrage` pour utiliser le texte des questions comme labels, en lisant la config.

**Amélioration #5 — `web.py` : double sérialisation dans `_project_dict`**
`model_dump()` sérialise déjà les modèles imbriqués ; la ligne `d["structure"] = project.structure.model_dump()` est redondante. Supprimer.

**Amélioration #6 — `project_manager.py` : exposer `get_corrections_dir()` public**
`web.py` accède à `project_manager._project_dir` (fonction "privée") pour construire le chemin des corrections. Exposer une fonction publique `get_corrections_dir(project_id)` dans `project_manager.py` et mettre à jour l'appelant.

# Prompt 34 — Fix : lignes de titre de catégorie écrasées par les réponses

## Problème

En comparant `_work/2.2. Annexe 2 CCAP Conformité RGPD_2026-02-AO-AC-AMOA2.xlsx` (original) et `_work/PAS_rempli_c3a07f7a.xlsx` (fichier généré), les lignes 25, 26, 33 et 42 sont corrompues :

- **Original** : ces lignes contiennent des titres de section/sous-section (`"Sécurité informatique"`, `"Politique"`, `"Pratique"`, texte intro "Stockage de fichiers…")
- **Rempli** : ces lignes contiennent `"Oui"` + un commentaire — les titres ont été écrasés

## Cause racine

Ces lignes ont leur colonne de réponse (col B) fusionnée avec col A via une cellule fusionnée :
- A25:C25, A26:B26, A33:B33, A42:B42

Quand `_write_cell` essaie d'écrire dans col B, elle détecte une `MergedCell`, remonte au coin supérieur gauche de la plage fusionnée (col A) et écrase le titre de section.

Par ailleurs, `read_questions` incluait ces lignes comme de vraies questions (puisqu'elles ont du texte non vide en col A).

## Correctifs

### 1. `system_structure.txt`
Ajout d'une note expliquant le concept de lignes de séparation de catégorie (cellules fusionnées, non-questions).

### 2. `parser_xlsx.py` — `read_questions`
- Nouvelle fonction `_merged_header_rows(ws, col_r_idx)` : retourne l'ensemble des numéros de lignes où col_response est une slave MergedCell (top-left dans une colonne différente).
- Stratégie **deux passes** pour ne pas exploser la mémoire :
  - Pass 1 : chargement complet (sans `read_only`) uniquement pour lire les plages fusionnées → `del wb; del ws` immédiatement après.
  - Pass 2 : chargement `read_only=True` (streaming) pour l'extraction réelle des questions.
- Lignes détectées comme en-têtes de section sont sautées avant extraction.
- **Fix OOM** : le premier `read_questions` sans `read_only` avait poussé le processus à >1.5 GB de RSS sur le droplet 2 GB → l'approche deux passes rétablit la consommation mémoire minimale.

### 3. `parser_xlsx.py` — `write_responses`
- Appel à `_merged_header_rows` avant la boucle d'écriture.
- Toute ligne dans `header_rows` est sautée (safeguard de second niveau).

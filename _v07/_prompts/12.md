# Prompt 12 — Base de connaissances (corpus)

## Contexte

Nouvelle "side feature" distincte du flux principal "analyser un nouveau PAS".
Permet d'alimenter la base de connaissances du LLM avec des questionnaires de sécurité déjà remplis.

## Prompts de la conversation

### 1
> We are going to implement a "side feature": this new feature must be distinct from the current "upload and anonymize xlsx file". This new "side feature" will deal with the "already filled security questionnaires". These xlsx have already been answered, and must serve as input for the LLM, to take example on. Each xlsx must be tagged with metadata. The metadata must be defined by the user through a wizard. The questions are the usual ones. Read the project files and try to guess what are those questions.

**Réponse :** Analyse des specs. Les "questions habituelles" = questions de cadrage de `questions.txt`, mappées aux champs metadata de F-REF-02 :
`type_prestation`, `nb_etp`, `activites`, `expertise_atlassian`, `hebergement_donnees`, `cloud_provider`, `sous_traitance_rgpd`, `lieu_travail`, `agences`, `poste_travail`, `connexion_distante`, `secteur_client`, `date_remplissage`, `tags_supplementaires`.

### 2
> j'ai corrigé (web/ manquant). Nous pouvons revenir au sujet principal. La liste de questions/métadonnées me convient. Mais avant de coder cette fonctionnalité, commence par créer un menu, avec la fonctionnalité existante pour analyser un nouveau PAS et cette fonctionnalité pour alimenter le LLM avec des exemples déjà remplis.

**Réponse :** Ajout d'une navigation par onglets dans `private.html` :
- Onglet "Analyser un nouveau PAS" (fonctionnalité existante)
- Onglet "Base de connaissances" (placeholder)
- Fonction `switchTab(name)`, CSS `.nav-tab` / `.tab-panel`

### 3
> Maintenant tu peux implémenter la fonctionnalité d'upload d'un PAS existant. ATTENTION : le wizard qui pose les questions sera utilisé par les 2 fonctionnalités ! Le code doit être mutualisé.

**Réponse :** Implémentation complète :
- `questions.txt` : ajout directive `KEY:` sur chaque question + 2 nouvelles questions (`cloud_provider` conditionnel sur "Cloud", `secteur_client`)
- `config.py` : parsing `KEY:`, fallback `q{n}`
- `web.py` : `GET /api/questions`, `POST /api/corpus`, `POST /api/corpus/{id}/metadata`, `GET /api/corpus`, `DELETE /api/corpus/{id}`
- `private.html` : classe `QuestionWizard` partagée, flux corpus (upload → wizard → infos → save), liste corpus

### 4
> I have a question about the implementation. About "base de connaissances"; is there an anonymization step?

**Réponse :** Oui — les fichiers corpus sont envoyés au LLM comme références → doivent être anonymisés (R-02). La spec dit "en amont" mais l'outil doit inclure cette étape. Proposition d'ajouter l'anonymisation entre le wizard et les infos complémentaires.

### 5
> Oui s'il-te-plaît

**Réponse :** Ajout de l'étape anonymisation dans le flux corpus :
- `anonymizer.py` : `extract_metadata_docx` et `anonymize_docx` (python-docx, remplacement dans paragraphes + tableaux, strip core properties)
- `web.py` : `GET /api/corpus/{id}/anon-suggestions`, `POST /api/corpus/{id}/anonymize` (xlsx et docx)
- `private.html` :
  - `addKwRow(containerId, …)` et `getKwPairs(containerId)` **partagées** entre PAS et Corpus
  - `pasAddRow` réduit à un wrapper
  - `corpus-anon-section` insérée entre wizard et extra
  - `corpusLoadAnonSuggestions()`, `corpusDoAnonymize()`

### 6
> How are persisted the metadata server side?

**Réponse :** Cycle de vie de `metadata.json` :
1. Après upload : `{"filename": "...", "format": "xlsx"}`
2. Après anonymisation : fichiers `anonymized.{ext}` et `anonymized_map.json` créés (metadata.json non modifié)
3. Après wizard + save : metadata.json réécrit complet avec tous les champs cadrage + date + tags

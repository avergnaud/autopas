# Prompt 18 — Corrections des déviations mineures

Ce prompt corrige cinq déviations entre l'implémentation actuelle (_v09) et les spécifications.
Il ne touche pas au pipeline de génération (prévu dans un baby step ultérieur).

---

## 1. Renommage de l'endpoint upload

**Fichier concerné : `app/api/web.py`**

Remplacer l'endpoint `POST /api/upload` par `POST /api/projects`.

C'est la convention REST correcte : créer un projet = `POST` sur la collection `/api/projects`.
Le frontend (`private.html` / `app.js`) devra être mis à jour en conséquence pour appeler `/api/projects`
au lieu de `/api/upload`.

---

## 2. Formats acceptés — uniquement xlsx (mise à jour de la spéc)

**Fichiers concernés : `SPECIFICATIONS_FONCTIONNELLES.md`, `SPECIFICATIONS_TECHNIQUES.md`**

La V1 ne supporte que le format `.xlsx` pour le wizard "Analyser un nouveau PAS".
Mettre à jour les spécifications pour refléter ce choix :
- Section 2.1 "Inclus dans la V1" : formats d'entrée = `.xlsx` uniquement.
- Section 6.1 F-PARSE-01 : la détection de format est supprimée, le format est toujours xlsx.
- Section 4.1 tableau des endpoints : `POST /api/projects` — la description ne mentionne plus docx.

Le format `.docx` reste supporté pour le corpus (wizard "Base de connaissances"), car l'anonymisation
docx est déjà implémentée et utilisée dans ce contexte.

## 3. Anonymisation obligatoire (mise à jour de la spéc)

**Fichiers concernés : `SPECIFICATIONS_FONCTIONNELLES.md`**

L'étape d'anonymisation est **obligatoire** dans le wizard "Analyser un nouveau PAS".
Mettre à jour les spécifications :
- F-WEB-06 : supprimer la mention "Cette étape est optionnelle" et le bouton "Passer cette étape".
- Le bouton s'appelle simplement "Anonymiser et continuer".

Dans le code (`app/api/web.py`), le comportement actuel (retour 400 si aucun mot-clé fourni) est correct

## 4. État projet — implémentation de project.json et machine à états

**Fichiers concernés : `app/api/web.py`, nouveau `app/services/project_manager.py`**

### 4.1 Créer `app/services/project_manager.py`

Ce module gère la lecture et l'écriture du fichier `data/projects/{id}/project.json`.

Il expose au minimum :

```python
def create_project(project_id: str, filename: str, user_email: str) -> dict
    """Crée project.json avec status="created"."""

def load_project(project_id: str) -> dict
    """Lit et retourne project.json. Lève FileNotFoundError si absent."""

def save_project(project_id: str, data: dict) -> None
    """Écrit project.json (remplace entièrement)."""

def update_project(project_id: str, **fields) -> dict
    """Met à jour des champs spécifiques de project.json et retourne le dict mis à jour."""

def list_projects(user_email: str) -> list[dict]
    """Liste tous les projets de l'utilisateur (filtre sur user_email)."""

def recover_stale_projects() -> None
    """Au démarrage : passe en status="error" les projets bloqués en status="generating"."""
```

### 4.2 Structure de project.json

```json
{
  "id": "...",
  "created_at": "2026-03-08T10:00:00Z",
  "updated_at": "2026-03-08T10:05:00Z",
  "user_email": "user@fournisseur.com",
  "status": "created",
  "original_filename": "Questionnaire_SSI.xlsx",
  "format": "xlsx",
  "structure": null,
  "cadrage": null,
  "anonymization": null,
  "verbosity_level": 2,
  "progress_step": null,
  "error_message": null
}
```

### 4.3 Machine à états

Les valeurs possibles de `status` sont :

```
created → anonymizing → generating → completed
                                   → error
```

- `created` : projet créé après upload, working.xlsx disponible.
- `anonymizing` : anonymisation effectuée, anonymized.xlsx + anonymized_map.json disponibles,
  structure détectée et confirmée (structure.json disponible), cadrage en cours ou terminé.
  Ce statut couvre toute la phase de préparation avant la génération.
- `generating` : génération en cours en arrière-plan (BackgroundTask).
- `completed` : génération terminée, output.xlsx et attention.md disponibles.
- `error` : une étape a échoué (message dans `error_message`).

### 4.4 Mise à jour des endpoints dans `app/api/web.py`

Chaque endpoint doit lire/écrire `project.json` via `project_manager` :

- `POST /api/projects` (ex-`/api/upload`) : appelle `create_project()`, retourne l'id et le status.
- `POST /api/projects/{id}/anonymize` : après succès, appelle `update_project(id, status="anonymizing", anonymization={"mappings": mapping})`.
- `POST /api/projects/{id}/structure` : appelle `update_project(id, structure=body.model_dump())`.
- `GET /api/projects` : appelle `list_projects(user_email)`, retourne la liste.
- `GET /api/projects/{id}` : appelle `load_project(id)`, vérifie `user_email`, retourne le dict.

### 4.5 Vérification des préconditions

Chaque endpoint vérifie que le projet est dans le bon état avant d'agir :

- `POST /api/projects/{id}/anonymize` : vérifie que `anonymized.xlsx` n'existe pas déjà
  (ou qu'on est bien en `created` ou `anonymizing`).
- `POST /api/projects/{id}/detect-structure` : vérifie que `anonymized.xlsx` existe.
- `POST /api/projects/{id}/structure` : vérifie que le projet existe.
- (Future) `POST /api/projects/{id}/generate` : vérifiera que `status == "anonymizing"`.

### 4.6 Récupération au démarrage

Dans `app/main.py`, appeler `recover_stale_projects()` dans la fonction `lifespan` au démarrage,
avant le `yield`. Cela passe en `error` tous les projets dont `status == "generating"` au redémarrage
du serveur.

### 4.7 Nouveaux endpoints à ajouter

```
GET  /api/projects           → liste les projets de l'utilisateur courant
GET  /api/projects/{id}      → détail d'un projet (vérifie l'appartenance)
```

## 6. Prompts système — externaliser dans des fichiers

**Fichiers concernés : `app/services/structure_analyzer.py`, nouveau `data/config/prompts/`**

### 6.1 Créer le répertoire et les fichiers prompts

Créer `data/config/prompts/` avec trois fichiers conformément à la spec (§11) :

- `data/config/prompts/system_structure.txt` — prompt système pour l'analyse de structure xlsx.
  Contenu : déplacer le prompt actuellement codé en dur dans `structure_analyzer.py`
  (variables `system` et `user` template).
- `data/config/prompts/system_response.txt` — prompt système pour la génération des réponses
  (contenu : §11.1 de SPECIFICATIONS_TECHNIQUES.md).
- `data/config/prompts/system_attention.txt` — prompt système pour les points d'attention
  (contenu : §11.2 de SPECIFICATIONS_TECHNIQUES.md).

### 6.2 Modifier `structure_analyzer.py`

Ajouter une fonction `_load_prompt(name: str) -> str` qui lit le fichier
`data/config/prompts/{name}` depuis `BASE_DIR`.

Remplacer les chaînes `system` et `user` codées en dur par des appels à `_load_prompt()`.

Le template de prompt `system_structure.txt` doit contenir un marqueur `{preview}` que la fonction
`detect_xlsx_structure()` remplace par le contenu formaté avant l'appel à Claude.

### 6.3 Gestion de l'absence du fichier

Si un fichier prompt est absent, lever une `RuntimeError` explicite (comme pour `ANTHROPIC_API_KEY`),
afin d'avoir un message d'erreur clair plutôt qu'un crash opaque.
